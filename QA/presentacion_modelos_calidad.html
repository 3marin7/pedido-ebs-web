Modelo Histórico · 1991–2011

ISO/IEC 9126

Predecesor del 25010. Fue el estándar dominante durante 20 años. Define 6 características fundamentales.

⚙️
Funcionalidad
Idoneidad, exactitud, interoperabilidad, seguridad, conformidad
💪
Confiabilidad
Madurez, tolerancia a fallos, capacidad de recuperación
👁️
Usabilidad
Comprensibilidad, aprendizaje, operabilidad, atracción
🚀
Eficiencia
Comportamiento temporal y utilización de recursos
🛠️
Mantenibilidad
Analizabilidad, cambiabilidad, estabilidad, evaluabilidad
🌍
Portabilidad
Adaptabilidad, instalabilidad, coexistencia, reemplazabilidad

El ISO 9126 fue reemplazado por ISO/IEC 25010 en 2011, que agrega Seguridad y Compatibilidad como características independientes.

Madurez del Proceso · CMMI

CMMI — Capability Maturity Model

Mide la madurez del proceso de desarrollo, no el producto. Evalúa la organización en 5 niveles.

1
Inicial — Caótico
Procesos sin documentar, éxito depende de personas individuales
2
Gestionado — Planificado
Procesos planificados, controlados y con métricas básicas
3
Definido — Estandarizado ← Tu proyecto apunta aquí
Procesos documentados y estandarizados en toda la organización
4
Cuantitativamente Gestionado
Medición y control estadístico de calidad y rendimiento
5
En Optimización — Mejora Continua
Mejora continua basada en análisis cuantitativo de datos
Modelo Histórico · 1977

Modelo de McCall

Uno de los primeros modelos formales de calidad. Define 3 perspectivas del producto.

🖥️
Operación del Producto
  • Corrección
  • Confiabilidad
  • Eficiencia
  • Integridad
  • Usabilidad
🔍
Revisión del Producto
  • Mantenibilidad
  • Flexibilidad
  • Capacidad de prueba
🚀
Transición del Producto
  • Portabilidad
  • Reusabilidad
  • Interoperabilidad

McCall fue revolucionario en 1977 por ser el primer modelo que clasificó la calidad en factores medibles, sentando las bases de todos los estándares posteriores.

Modelo Histórico · 1978

Modelo de Boehm

Expande a McCall añadiendo consideraciones de hardware y una jerarquía más detallada.

🧑‍💻
Utilidad General
Portabilidad, confiabilidad, eficiencia, usabilidad
🔧
Utilidad para Mantenimiento
Verificabilidad, comprensibilidad, modificabilidad
🌐
Utilidad de Transición
Interoperabilidad, reusabilidad, portabilidad de dispositivo

Boehm introdujo el concepto de utilidad primaria del usuario y fue el primero en incluir factores relacionados con el hardware en el modelo de calidad.

Hewlett-Packard · RUP

FURPS+

Modelo de HP, muy usado en metodología RUP (Rational Unified Process). Clasifica requisitos de calidad en 5 dimensiones + restricciones.

F
Functionality
Requisitos funcionales del sistema
U
Usability
Estética, consistencia, documentación
R
Reliability
Frecuencia de fallos, recuperación
P
Performance
Velocidad, exactitud, throughput
S
Supportability
Mantenibilidad, adaptabilidad, testabilidad

El + agrega 4 restricciones adicionales: Diseño (restricciones arquitectónicas), Implementación (lenguajes, BD), Interfaz (integraciones), Físicas (hardware, materiales).

Estándares IEEE

Estándares IEEE

El Instituto de Ingenieros Eléctricos y Electrónicos define estándares de proceso para la industria del software.

📋
IEEE 730 — SQA Plan
Define los requisitos para un Plan de Aseguramiento de Calidad de Software (SQA).

Cubre: documentación, revisiones, auditorías, gestión de problemas y herramientas.
🔬
IEEE 1012 — V&V
Estándar para Verificación y Validación de software.

Verificación: ¿Estamos construyendo el sistema correctamente?
Validación: ¿Estamos construyendo el sistema correcto?

Tu proyecto QA/01_PLAN_ASEGURAMIENTO_CALIDAD.md ya implementa los requerimientos de IEEE 730 al tener plan de QA con métricas, herramientas y actividades definidas.

Guía de Decisión

¿Cuál modelo usar?

Cada modelo responde a una necesidad diferente. Aquí la guía rápida de selección.

SituaciónModelo recomendadoPor qué
Evaluar calidad del producto web/app ISO/IEC 25010 Estándar vigente más completo
Mejorar el proceso de desarrollo CMMI Evalúa madurez organizacional
Proyecto académico o institucional ISO 25010 + IEEE 730 Combinación reconocida formalmente
Empresa con múltiples proyectos CMMI Nivel 3+ Estandarización entre proyectos
Documentar requisitos no funcionales FURPS+ Estructura natural para RNF
Definir V&V formalmente IEEE 1012 Estándar específico de V&V
Proyecto legacy o mantenimiento ISO 9126 Más sencillo, igual de válido
